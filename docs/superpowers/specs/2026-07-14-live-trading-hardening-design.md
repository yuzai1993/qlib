# Live Trading 实盘关键链路加固设计

日期：2026-07-14  
状态：已获用户批准实施（采用“实盘关键链路加固”方案）

## 1. 背景与目标

当前 live trading 已覆盖信号发布、QMT 执行、回执导入、持仓现金账本、监控、交易费用与公司行为，但 review 发现若干会直接造成错单或错账的问题：同日批次订单 ID 冲突、回执缺少订单级核验、部分成交增量金额错误、QMT 买单未预占现金、执行中批次无法在重启后恢复，以及分红权益、到账日和红利税的会计口径不准确。

本轮目标是在不引入完整券商清算系统的前提下，使以下性质成立：

1. 不同批次的订单与回执不可串批。
2. 非法、越界或与原订单不一致的回执不得改变持仓和现金。
3. 多次累计成交回执只按真实新增数量、成交额和费用记账。
4. QMT 重启后不会因为内存状态丢失而重复提交已知订单。
5. 同一批买单使用一个可用资金预算，并逐单预占成交金额和费用。
6. 实盘信号只能使用指定交易日的最新特征，且发布到下一个真实交易日。
7. 分红权益按股权登记日持仓确定；现金、送转股分别按派息日和上市日到账。
8. 红利税 20% 作为保守准备金，不伪装成券商已扣税。
9. 迁移与重跑保持幂等，现有数据库可原位升级。

不在本轮范围：逐笔持仓 lot、卖出 FIFO 和按每一 lot 自动计算最终 20%/10%/0% 红利税；该能力需要券商资金流水或完整成交历史，留作后续增强。

## 2. 订单身份与协议

### 2.1 全局唯一订单 ID

`client_order_id` 改为：

```text
YYYYMMDD + batch_seq(3) + order_seq(3) + side(1)
```

例如 `20260715002003B` 表示 2026-07-15 第 2 批第 3 个买单。总长 15，低于 QMT remark 的 24 字符限制。

`OrderPlanner.plan` 显式接收 `batch_seq`；发布脚本的 `--seq` 同时参与 `batch_id` 和订单 ID。单批内部仍校验 ID 唯一。

### 2.2 数据库身份

`fills` 与 `signal_orders` 改用 `(batch_id, client_order_id)` 联合主键。旧表在 `LiveRecorder` 初始化时原位迁移：创建新表、复制数据、替换旧表、重建索引。新 ID 本身全局唯一，联合主键是第二层防御，并兼容历史重复 ID。

`fills` 新增 `applied_amount`，保存已经作用到账本的累计成交额。旧数据按 `applied_qty * avg_price` 回填；这只能还原旧记录的最终状态，但不会重复调整已有现金。

### 2.3 回执一致性校验

结构校验增加：

- `requested_qty`、`filled_qty` 必须为非负整数；
- `filled_qty <= requested_qty`；
- 有成交时 `avg_price > 0`，无成交时价格不得为负；
- 股票代码格式合法。

`LiveRecorder.apply_fill` 在事务内读取原始批次与计划订单，并校验：

- 批次存在，模式与批次一致；
- 订单存在，股票和方向一致；
- QMT 因可卖数量或现金不足缩单时，`requested_qty` 可小于计划数量，但必须大于 0、为整手且不可超过计划数量；
- 同一订单后续回执的 `requested_qty` 不得变化；
- 累计成交数量和累计成交额不得倒退；
- 已应用成交数量不可超过请求数量。

任何失败均整条拒绝，不更新 `fills`、持仓、现金或费用。

## 3. 成交金额、持仓成本与费用

每次回执先计算：

```text
cumulative_amount = filled_qty * avg_price
delta_qty          = filled_qty - applied_qty
delta_amount       = cumulative_amount - applied_amount
```

买入持仓成本使用 `delta_amount`，而非 `delta_qty * 最新累计均价`；卖出现金同样使用 `delta_amount`。这可以正确处理 `100 股均价 10` 后更新为 `200 股累计均价 11` 的场景：第二次新增成交额为 1200，而不是 1100。

费用仍按订单累计成交额重算，再减去 `applied_fee`，因此最低佣金只计一次。费用配置在加载时校验为有限非负数，`side` 只能为 `BUY` 或 `SELL`。

## 4. QMT 执行可靠性

### 4.1 processing 生命周期与恢复

信号被领取后继续保留在 `processing/`，只在批次完成并写出回执 `.done` 后移入 `archive/`。

每个执行中批次在 `state/active_<batch_id>.json` 保存：阶段、已提交订单、最新回执、剩余买入预算等。状态文件通过临时文件加原子替换写入。QMT 初始化时优先恢复 `processing/`：

- 已在 `processed_batches.txt` 的批次只补归档，不再执行；
- 未完成批次加载信号与状态后继续轮询；
- 本地已标记提交的订单绝不自动重提；
- 提交前先持久化“将提交”状态，崩溃窗口内选择漏单而非重复下单，这是实盘更安全的失败方向；
- 恢复后仍通过 QMT remark 查询真实订单状态。

### 4.2 买入资金预占

进入 BUY 阶段时只读取一次 QMT 可用资金，存为批次 `remaining_cash`。每个订单按实际提交限价计算：

```text
reserved = quantity * price + max(commission, min_commission) + transfer_fee
```

若预算不足，按整手向下缩单；仍不足一手则跳过。提交后立即从 `remaining_cash` 扣减并持久化，不依赖 QMT 账户缓存是否及时刷新。

### 4.3 收盘处理

撤单后继续轮询到 finalize 时间；最终回执使用最后可见累计成交数量。未能确认终态的订单标记 `EXPIRED`/`PARTIAL`，同时保留批次与 QMT remark，便于盘后对账发现差异。

## 5. 发布安全与交易日

### 5.1 严格特征日期

`SignalGenerator.predict` 增加 `allow_stale` 参数。paper 场景默认保持兼容；live 发布显式传 `False`。如果目标 `signal_date` 不在特征索引中，发布立即失败，不再用更早日期静默替代。

### 5.2 下一个交易日

夜间发布不能使用自然日 `+1`。新增小型日期解析脚本，通过 Tushare `trade_cal` 查询今天之后首个开市日；缺少 token、查询失败或没有结果时拒绝发布。手工传入日期时仍支持显式覆盖。

cron 继续显式传 `--mode LIVE`，同时保留 Mac 环境变量 `LIVE_TRADING_CONFIRM=YES` 与 QMT `LIVE_OK_<trade_date>` 双开关。

## 6. 公司行为子账

### 6.1 事件与权益

在除息日通过 Tushare `dividend` 获取 `record_date`、`ex_date`、`pay_date`、`div_listdate`、`cash_div_tax`、`stk_div`。事件只在 `div_proc=实施` 时处理。

新增 `corporate_actions` 表，以稳定事件键幂等记录：

- 股票代码与各关键日期；
- 股权登记日权益股数；
- 税前现金红利、红利税准备、送转股数；
- 应收确认、现金到账、送转股上市等状态。

权益股数只读取 `position_snapshot(record_date, stock_code)`。若登记日快照不存在，系统不得用除息日当前持仓猜测，而是产生 `CORP_ACTION_ENTITLEMENT_MISSING` 告警，等待人工核验。

### 6.2 应收、现金和送转股

- 除息日：确认现金红利应收和红利税准备，不增加可交易现金；送转股记为待上市股数。
- 派息日：应收转入现金，净值不重复变化。
- 红股上市日：待上市股数转入普通持仓并摊薄平均成本。

监控快照增加 `receivables`、`pending_market_value` 和 `tax_provision`，总资产口径为：

```text
cash + listed_market_value + receivables + pending_market_value - tax_provision
```

待上市送转股按当日该股票收盘价估值；它不进入 `LiveRecorder.get_positions()`，因此不会提前生成卖单。

### 6.3 红利税准备

默认按税前红利的 20% 确认 `tax_provision`，名称和页面均显示为“红利税准备”，不写成真实已扣税。最终税率仍由持有期决定；未来导入券商真实扣税时，结算准备并把差额计入当日损益。

公司行为事件、应收/准备余额、现金变化、送转股变化与流水必须在同一个 SQLite 事务中更新，避免流水成功但持仓失败后无法重试。

## 7. 资金流水约束

资金流水类型与符号进行校验：

- `DEPOSIT` 必须为正；
- `WITHDRAW` 必须为负；
- 公司行为流水由内部事务生成，普通 CLI 不允许伪造 `BONUS_SHARES`；
- 手工校正必须提供备注。

外部出入金只包含 `DEPOSIT` 与 `WITHDRAW`。`CORRECTION` 不再默认视为外部资金，避免交易费、税费或成交对账差额被错误地从收益中剔除。需要调整初始本金时应使用明确的存取款流水。

## 8. 测试与迁移验证

先补失败测试，再实现：

1. 同日两个批次生成不同订单 ID，两个批次订单和回执均可共存。
2. 错批次、错股票、错方向、超量、倒退回执全部拒绝且账本不变。
3. 变动累计均价的部分成交正确更新现金、成本和费用。
4. QMT 重启从 processing 恢复，不重提已提交订单。
5. 多笔买单在缓存不刷新的情况下也不会超过本地预算。
6. 周五解析得到下周首个开市日；查询失败时不发布。
7. live 严格拒绝过期特征。
8. 除息日买入/卖出不改变登记日权益；派息日和上市日分别结算。
9. 公司行为重跑幂等，故障回滚后可以完整重试。
10. 红利税准备影响净值但不提前减少可交易现金。
11. 旧数据库迁移后历史行、现金和持仓保持不变。

完成后运行 `pytest -q tests/live_trading`，并针对共享目录使用临时目录做一遍端到端发布、领取、回执、导入演练。

## 9. 上线与运行环境清理

代码验证通过后：

1. 清空共享目录 `archive/`、`inbox/`、`outbound/`、`processing/`、`logs/`、`state/` 中的旧文件，保留目录结构。
2. 删除旧的 2026-07-15 信号；它使用旧订单 ID 协议，不得执行。
3. 将验证后的 QMT bridge 覆盖到共享目录 `strategy/qmt_signal_bridge.py`。
4. 不删除本地 live SQLite；本地资金和历史是独立账本，除非用户另行明确授权重置。
5. crontab 只保留 live 链路及其必要的数据更新依赖，统一注释为 `live:`。当前 Tushare→bin 日更属于 live 特征与收盘价的必要依赖，保留。
6. 重新生成下一交易日 LIVE 信号前，再检查本地账本现金、持仓、`LIVE_TRADING_CONFIRM` 和 QMT 当日 `LIVE_OK` 开关。

## 10. 回滚

代码通过 Git commit 回滚。数据库迁移前自动生成同目录时间戳备份；若迁移或冒烟测试失败，停止发布并恢复备份。共享目录清理后不恢复旧信号，避免旧协议信号重新进入 QMT。
