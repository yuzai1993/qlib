# QMT 动态报单保护价设计

**日期：** 2026-07-17
**状态：** 已确认范围，待书面审阅

## 1. 背景

当前实盘信号在 Mac 侧按前收盘价生成固定边界：买入为前收盘价上浮 1%，卖出为前收盘价下浮 1%。QMT 侧虽然会按实时最新价加入 0.3% 成交缓冲，但仍受上述固定边界封顶或托底。

该设计会系统性阻止正常调仓：当股票上涨超过约 1% 时，买入价格可能低于卖盘；当股票下跌超过约 1% 时，卖出价格可能高于买盘。尾盘有限执行窗口进一步放大未成交风险。

## 2. 目标与非目标

### 目标

- 首次报单价格锚定实时对手方盘口，提高正常行情下的成交概率。
- 移除前收盘价 ±1% 对实时报单价格的硬封顶和硬托底。
- 保留 0.3% 的盘中成交缓冲。
- 使用 QMT 当日涨停价和跌停价约束最终报单，避免价格越界。
- 保持信号文件协议兼容，不要求 Mac 与 Windows 同时升级协议版本。

### 非目标

- 本次不实现未成交订单的撤单、追价和重新报单。
- 本次不改变先卖后买、资金预留、14:56 撤单和 14:57 收尾流程。
- 本次不修改回测成交模型或策略选股逻辑。
- 本次不保证涨停买入或跌停卖出一定成交；没有对手盘时限价无法创造流动性。

## 3. 方案比较

### 方案 A：对手方盘口动态限价并钳制涨跌停价（采用）

买入锚定卖一价，卖出锚定买一价；对应盘口缺失时退回最新价。加入实时 0.3% 缓冲后，再用 QMT 的 `UpStopPrice` / `DownStopPrice` 钳制。

优点是报单通常可立即穿过一档盘口，同时仍有明确的合法价格边界。改动集中在价格读取和计算，不改变订单状态机。

### 方案 B：仅移除前收盘价边界，继续使用最新价 ±0.3%

实现最小，但最新成交价不一定等于当前对手方报价。在价差扩大或最新成交滞后时，成交率低于方案 A。

### 方案 C：动态限价加循环撤单追价

理论成交率最高，但需要新增撤单确认、剩余数量、部分成交、重启恢复和重复报单防护。该方案显著扩大实盘状态机风险，留作独立后续改动。

## 4. 详细设计

### 4.1 行情快照

QMT 侧继续通过 `ContextInfo.get_full_tick([stock_code])` 获取单只股票快照，读取：

- `askPrice[0]`：卖一价；
- `bidPrice[0]`：买一价；
- `lastPrice`：最新成交价，作为对手方盘口缺失时的回退参考价。

单个字段必须能够转换为大于零的有限浮点数，否则视为无效。空列表、零值、缺失字段和异常类型均不得使桥接策略崩溃。

### 4.2 当日价格边界

通过 `ContextInfo.get_instrumentdetail(stock_code)` 获取合约详情：

- `UpStopPrice`：当日涨停价；
- `DownStopPrice`：当日跌停价。

边界字段只有在能够转换为大于零的有限浮点数时才参与钳制。详情接口缺失、抛错或返回无效字段时，记录日志并继续使用实时动态价格；交易所仍会执行最终有效申报校验。

### 4.3 报单价格

买入参考价按以下顺序选择：有效卖一价、有效最新价。卖出参考价按以下顺序选择：有效买一价、有效最新价。

原始报单价为：

```text
BUY  raw_price = reference_price * (1 + INTRADAY_BUY_SLIPPAGE)
SELL raw_price = reference_price * (1 - INTRADAY_SELL_SLIPPAGE)
```

当前两个盘中缓冲参数均保持 0.003。

最终价格先按 A 股 0.01 元最小变动单位四舍五入，再应用可用的当日边界：

```text
BUY  effective_price = min(rounded_raw_price, UpStopPrice)
SELL effective_price = max(rounded_raw_price, DownStopPrice)
```

前收盘价生成的信号 `limit_price` 不再限制有效实时报单价。

### 4.4 降级行为

- 对手方一档缺失但 `lastPrice` 有效：使用最新价计算动态报单价。这允许涨停买单或跌停卖单以边界价格进入队列，但不保证成交。
- 整个实时快照缺失或不存在有效参考价：退回信号中的 `limit_price`，保持当前故障降级行为，避免用零价或异常价报单。
- 合约详情不可用：不使用涨跌停钳制，记录错误后按动态价格报单；若价格无效，QMT/交易所拒单并由现有回执流程记录。

### 4.5 协议与配置兼容

`SignalOrder.limit_price`、`buy_slippage` 和 `sell_slippage` 本次保留：

- `limit_price` 继续作为实时行情完全不可用时的故障回退价；
- 信号 JSONL schema 不升级；
- 既有在途信号和 Windows QMT 桥接文件仍可被读取；
- Mac 侧 `OrderPlanner` 的数量、优先级、前收盘价校验和订单 ID 行为不变。

实现中的注释、日志和文档需将 `limit_price` 从“硬风险边界”改称“故障回退价”，避免产生错误运维预期。

## 5. 错误处理与可观测性

- `get_full_tick` 和 `get_instrumentdetail` 的异常不得中断整个批次。
- 行情或边界读取失败时记录股票代码和异常堆栈。
- 下单日志同时记录最终报单价与信号回退价，便于区分正常动态定价和故障降级。
- 既有 `ACCEPTED`、`REJECTED`、`EXPIRED`、`ERROR` 回执语义不变。

## 6. 测试设计

纯逻辑测试覆盖：

1. 买入使用卖一价加 0.3%，即使超过信号 `limit_price` 也不再被封顶。
2. 卖出使用买一价减 0.3%，即使低于信号 `limit_price` 也不再被托底。
3. 买入价格不超过 `UpStopPrice`。
4. 卖出价格不低于 `DownStopPrice`。
5. 卖一或买一缺失时回退到 `lastPrice`。
6. 实时参考价完全不可用时回退到信号 `limit_price`。
7. 合约详情缺失或异常时仍能使用动态价格，且不会使批次崩溃。
8. 既有买入资金预留使用最终动态报单价，防止超额下单。

验证范围包括 `tests/live_trading/test_qmt_bridge_logic.py` 和完整的 `tests/live_trading` 测试集。

## 7. 验收标准

- 前收盘价 10 元、实时卖一价 10.50 元的买单，不再被 10.10 元信号价封顶。
- 前收盘价 10 元、实时买一价 9.50 元的卖单，不再被 9.90 元信号价托底。
- 接近涨跌停时，最终报单价分别不高于或不低于 QMT 提供的当日边界。
- 无实时行情时仍保持可预测的 `limit_price` 回退。
- 未引入撤单追价、协议升级或回测逻辑改动。
- 所有相关自动化测试通过。
