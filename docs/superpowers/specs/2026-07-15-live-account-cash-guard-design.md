# Live 资金账号纠正与资金查询防护设计

## 背景

2026-07-15 的有效批次 `20260715_csi300_topk10_003` 使用账号
`88813528`，但 QMT 界面显示的真实资金账号是 `8881352838`。批次的
10 个买单在 14:45 全部生成 `SKIPPED` 回执，消息均为
`insufficient reserved cash: 0.00`；没有 QMT 委托号、成交、费用或持仓变化。

桥接策略当前把以下三种状态都折叠成 `0.0`：账户查询异常、查询结果为空、
真实可用资金为零。因此错误账号没有被识别为配置错误，而被误报为资金不足。
同时，盘后监控把 `SKIPPED` 视为完整终态，却只把 `REJECTED/ERROR` 计入
执行失败率，导致 10/10 全跳过没有业务告警。

2026-07-16 的 `20260716_csi300_topk10_001` 已按错误账号发布到 inbox，
但 `LIVE_OK_2026-07-16` 不存在，尚无真实执行风险。

## 目标

1. 将发布链路使用的完整资金账号统一为 `8881352838`。
2. 保留 2026-07-16 错误批次的审计记录，发布正确账号的新批次并将旧批次标记为废弃。
3. QMT 无法取得账户资金时不提交、不伪装成零资金，并在交易窗口内持续重试。
4. 若收盘前始终无法取得账户资金，生成明确的 `ERROR` 回执。
5. 活动 LIVE 批次全部 `SKIPPED` 时产生 CRIT 告警。
6. 有活动批次时，16:00 的 postmarket 对账不因 qlib 当日日历尚未更新而提前退出。

## 方案选择

### 方案 A：只纠正账号

改动最小，但账户掉线、类型错误或 API 返回空列表时仍会被误判为可用资金为零，
无法防止同类事故。

### 方案 B：纠正账号并在组件边界 fail closed（采用）

在账号、QMT 查询、回执和监控四层同时保留明确语义。它能修复本次事故，也能让
未来的登录失效、账号类型错误和查询异常可观察、可告警。

### 方案 C：在 QMT 源码中硬编码账号

虽然能绕过错误 header，但会制造 Mac 配置和 Windows 源码两个事实源，后续更容易
漂移，因此不采用。

## 详细设计

### 1. 账号与批次替换

- 将 `~/.qlib_live_env` 中的 `QMT_ACCOUNT_ID` 改为 `8881352838`。
- 保持配置中的 `account_type=STOCK`；本次只纠正已由用户确认的完整资金账号。
- 使用序号 `002` 为 2026-07-16 发布新的 LIVE 批次。
- 新批次成功写入数据库和 inbox 后，才将 `001` 移入 archive，并在账本中标记
  `001 -> 002`。任一步失败均保留可恢复现场。
- 不创建 `LIVE_OK_2026-07-16`。

### 2. QMT 账户资金查询

`_get_available_cash(account_id)` 的返回语义改为：

- 非负 `float`：成功读取的真实可用资金，`0.0` 是合法值；
- `None`：API 异常、空列表、缺少 `m_dAvailable`、值不可转换、负数，或返回账号与
  请求账号不一致。

买入阶段遇到 `None` 时保持批次未提交状态，让后续 tick 重试，不把任何订单写成
`SKIPPED`。到 `FINALIZE_AT` 仍不可用时，未提交买单写成 `ERROR`，消息为
`account cash unavailable at close`，随后正常完成回执文件，避免批次永久卡住。

真实可用资金确实为 `0.0` 时继续沿用整手预算逻辑，订单可写为 `SKIPPED`，消息明确
表示资金不足。

### 3. 盘后监控

- 只对活动 LIVE 批次判断“全单跳过”。
- 当活动 LIVE 批次已有完整回执，且所有相关回执均为 `SKIPPED`、成交量均为 0 时，
  产生 `ALL_ORDERS_SKIPPED` / `CRIT`，消息包含去重后的回执原因。
- `SKIPPED` 仍是协议终态，因此 reconciliation 的 `missing=0` 语义不变；业务失败由
  新规则表达。
- postmarket 阶段若当日存在活动批次，即使 qlib 日历尚未更新，也继续执行回执对账；
  report 阶段仍严格要求当日行情，避免生成不可信快照。

### 4. 部署边界

Mac 侧代码、账号环境、数据库、共享目录和监控可由本次任务完成。QMT 内置策略使用
Windows 策略编辑器中的源码副本，因此修改后的 `qmt_signal_bridge.py` 必须在 Windows
端同步、编译并重启策略后才会生效。账号纠正后的 signal header 不依赖源码更新，
但在 Windows 同步完成前不得创建次日 LIVE_OK。

## 测试与验收

- QMT 纯逻辑测试覆盖：空账户返回 `None`、真实零资金仍为 `0.0`、查询失败不提交、
  收盘时生成明确 ERROR。
- 监控测试覆盖：全单 SKIPPED 触发 CRIT；正常成交不触发；16:00 日历陈旧但有活动
  批次仍执行 postmarket。
- 全量 live trading 回归必须通过。
- 正式验收确认：环境账号为 `8881352838`；2026-07-16 只有 `002` ACTIVE；inbox 只有
  正确 `002`；错误 `001` 在 archive；次日 LIVE_OK 不存在；监控 API 展示正确账号和批次。

## 非目标

- 不修改今日已经归档的原始回执或持仓账本。
- 不自动创建 2026-07-16 的 LIVE_OK。
- 不猜测或自动切换 `STOCK/CREDIT`；如 QMT 用完整账号查询仍为空，必须先在 QMT 界面
  核实账户类型和登录状态。
