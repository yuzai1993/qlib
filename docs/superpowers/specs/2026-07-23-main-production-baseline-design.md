# Main 生产基线整理设计

## 目标

从 `main` 创建独立整理分支，只迁移当前生产运行所需能力。整理完成后，该分支应能：

1. 执行当前 Tushare 数据采集、历史补数、指数成分更新和日度入库任务；
2. 训练并运行当前 CSI300 baseline 回测，也能复用已有训练产物执行免重训回测；
3. 执行当前 crontab 中的回执导入、数据更新、监控、信号发布、漏发补偿和晚间检查；
4. 运行相关单元测试、静态检查与运维排查工具；
5. 不包含尚未用于实盘的策略、参数搜索和研究实验。

## 方案比较

### 方案 A：整体合并 `dev` 后删除实验内容

优点是操作简单，缺点是很难证明已经删除所有实验依赖，且会把 55 个混合提交和无关历史带入 `main`。不采用。

### 方案 B：从 `main` 按生产能力选择最终文件快照

以 `dev` 的已验证最终状态为来源，按能力边界选择文件，并重建少量语义清晰的提交。对混合提交不做整提交 cherry-pick，从而避免意外引入实验文件。paper-trading 解耦结果以其独立提交的最终状态为准。此方案最容易审计和验证，采用。

### 方案 C：在 `dev` 上交互式变基并拆分 55 个提交

能够保留较细历史，但成本高、冲突面大，还容易影响当前包含未提交实验的工作区。不采用。

## 最终能力边界

### 1. 数据生产链路

保留当前生产使用的 Tushare collector 及其公共 collector 基础设施，包括：

- 日线和指数数据采集、标准化与 dump-bin 适配；
- `amount` 和前复权 `vwap` 产出；
- 历史缺失数据补齐；
- 前复权回溯完整性检查与指数覆盖检查；
- CSI300/CSI500/CSI1000 指数成分构建和日度更新；
- 日度包装脚本、分步骤失败处理、日志和微信告警；
- 新版 pandas 下 `ProcessInf` 的兼容修复；
- 上述功能的测试和操作说明。

不迁移未被当前日度任务调用的 JQData collector、Yahoo 实验修改、示例 notebook 和研究文档。

### 2. 通用回测框架

保留 YAML 配置驱动的训练/回测和免重训回测能力：

- 配置解析、校验、handler/model/dataset 构建；
- 训练记录、回测记录、指标提取和图表/报告归档；
- `limit_threshold` 表达式兼容；
- baseline 所需的 Qlib 评估与报告修复；
- 配置加载和 baseline 行为测试。

只保留 CSI300 生产 baseline 的训练配置和免重训配置。训练起点扫描、`n_drop` 扫描、动态仓位、择时、CSI500/CSI1000 对比配置、多种子、标签、集成和策略扫参均排除。

### 3. CSI300 生产 baseline

生产基线固定为：

- 股票池/基准：CSI300 / `SH000300`；
- 特征与模型：`Alpha158` + `LGBModel`；
- 训练区间：2006-01-02 至 2020-01-10；
- 策略：`TopkDropoutStrategy(topk=10, n_drop=2)`；
- 实盘配置引用同一份模型、handler、策略和交易参数；
- 回测配置保留训练运行和从 `20260711_223223_train_start_2006` 复用模型的运行方式。

模型二进制和本地 SQLite 账本仍按现有约定留在 Git 之外。配置和文档必须明确它们的本地路径与初始化要求。

### 4. 实盘生产链路

保留 `live_trading` 的完整生产能力：

- 配置加载、信号预测、TopkDropout 调仓意图和订单规划；
- 信号批次持久化、校验、原子发布和 QMT 文件桥；
- QMT 实时价格保护、资金检查、订单恢复和截止时间管理；
- 回执导入、批次隔离、持仓/现金/费用/分红记账；
- 快照、监控、告警、日报、预测查询和 Web 页面；
- 回补订单、股票名称、预测数据和现金流校正工具；
- 发布、catch-up、回执导入和监控 cron 包装脚本；
- Mac/QMT 两侧运维文档及所有相关测试。

`live_trading` 不再导入 `paper_trading`。共享的信号生成和 TopkDropout 订单计算移动到生产中立的位置，并由 live 脚本直接引用。整个模拟盘应用、数据库、网页、日报和调度不迁移。

## 数据流与调度验收

当前工作日 cron 链路按以下顺序验证：

1. 16:00 导入 QMT 回执，并执行盘后对账监控；
2. 16:30 运行 Tushare 日线增量、指数成分日更、VWAP 与复权巡检；
3. 20:30 生成快照并推送日报；
4. 21:30 预测下一交易日信号并原子发布 LIVE 批次；
5. 22:05 对漏发批次执行幂等 catch-up；
6. 22:15 检查下一交易日的有效发布批次。

真实外部动作不会在验证时触发。使用单元测试、临时目录、临时 SQLite、mock notifier、CLI `--help`/配置加载和 shell 静态检查证明链路可启动；模型数据存在时再运行 baseline 的最小真实回测冒烟测试。

## 提交结构

最终分支从 `main` 重建以下提交：

1. 生产数据采集、补数、巡检与日调度；
2. YAML 回测框架与 CSI300 baseline 配置；
3. 去除 paper-trading 依赖后的 live-trading 完整链路；
4. 运行说明、迁移清单和验收修复。

每个提交同时包含对应测试，避免代码与测试分离。

## 验证策略

- 对纳入的 Python 文件执行编译检查；
- 对所有 cron shell 脚本执行 `bash -n`；
- 运行数据 collector、回测配置、live-trading 和解耦模块的定向测试；
- 检查仓库中不存在 `live_trading -> paper_trading` 导入；
- 检查最终差异中不存在实验脚本、实验配置、动态策略或 paper-trading 应用；
- 使用本地 `~/.qlib/qlib_data/cn_data` 和已有 MLflow artifact 做 baseline 回测冒烟验证；
- 对受外部 SMB、QMT、Tushare token 和微信密钥限制的动作，验证到外部边界并明确记录未实际触发的部分。

## 风险控制

- 当前 `dev` 的未提交实验改动只存在于原工作区，不从工作树复制；所有来源均使用明确的 Git 提交或路径快照。
- 对混合提交采用路径级选择并用最终差异审计，避免包含实验文件。
- paper-trading 解耦任务必须先完成其测试；最终分支只引入已提交结果。
- `.gitignore` 保留模型结果、日志、账本、共享桥状态和密钥文件的忽略规则，防止生产数据进入版本控制。
